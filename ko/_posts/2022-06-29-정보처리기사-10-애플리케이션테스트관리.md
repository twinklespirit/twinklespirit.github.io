## Ch01 애플리케이션 테스트 케이스 설계


<!-- 솔직히 이건 필요 없는 부분이고 -->
### 1. 소프트웨어 테스트 케이스
1. 사용자가 요구한 기능, 성능, 사용성, 안정성을 충족하는지 확인
2. 잠재적인 결함이 있는지 확인


> 소프트웨어 테스트는 프로그램에 잠재된 (오류를 발견)하고 수정하여, 프로그램 실행 전에 (오류를 예방)하기위해 필요합니다.<br>
> 최종목표: 최종 사용자의 요구사항에 만족하여 제품의 신뢰도를 향상하자. (품질향상)

<br>

<!-- 그래도 기본적으로 알고 있어야 하는 부분임-->

#### 소프트웨어 테스트 기본 원칙

* 결함 존재 증명
    * '결함이 존재함을 밝혀라!'
    * 결함이 없음을 증명할 수는 없고, 결함을 줄이면 됨!

    <br>

* 완벽 테스팅은 불가능
    * '완벽하게는 절~대 못함!'
    * 이유: 시간과 자원 낭비 (무한 경로, 무한 입력값..)

    <br>

* 초기 집중
    * '조기 테스트를 추천한다!'
    * 이유: 시간과 비용이 절약
        * snowball effect: sw 개발 초기 체계적인 분석 및 설계가 수행되지 못하면 그 결과 프로젝트의 후반에 영향을 미치게 되어 비용이 커진다.

    <br>

* 결함 집중
    * 소프트웨어 전체 오류의 80%는 전체 모듈의 20% 이내에서 발견된다.
    * '파레토의 법칙'

    <br>

* 살충제 패러독스
    * 동일한 테스트 케이스에 의한 반복적 테스트는 새로운 버그를 찾지 못한다.
    * 정기적 리뷰와 개선 및 다른 시각에서의 접근이 필요하다.

    <br>

* 정황 의존성
    * 소프트웨어의 성격에 맞게 테스트를 실시해야 한다.
    * 정황과 비즈니스 도메인에 따라 테스트를 다르게 수행

    <br>

* 오류-부재의 궤변
    * 요구사항을 충족시키지 못한다면, 결함이 없다고 해도 품질이 높다고 볼 수 없다.

<br>

#### 소프트웨어 테스트 프로세스

1. 테스트 계획
2. 테스트 분석 및 디자인
3. 테스트 케이스, 스크립트, 시나리오 작성
4. 테스트 수행
5. 테스트 결과서 작성

<br>

* [테스트 시나리오](#3-소프트웨어-테스트-시나리오-작성)

* 테스트 슈트 (테스트 케이스를 실행환경에 따라 구분한 테스트 케이스의 집합)
* 테스트 결과서

#### 블랙박스 테스트

* 보통 코드를 외주했을 때 사용하는 '저수준의 테스트'
* '우리가 원하는 기능이 잘 작동하는지 확인하자'의 사용자관점

<br>

#### 화이트박스 테스트

* 구문 == 문장 커버리지
    * 프로그램 내의 모든 명령문을 적어도 한 번 수행 (구문 실행 개수로 계산)
* 결정 == 선택 == 분기 커버리지
    * 각 분기의 결정 포인트 내에 전체 조건식이 적어도 한 번은 T/F을 수행
* 조건 커버리지
    * 각 분기의 결정 포인트 내에 개별 조건식이 적어도 한 번은 T/F을 수행
* 조건결정 커버리지
    * 전체 조건식뿐만 아니라 개별 조건식도 T/F 한 번 결과가 되도록 수행
* 변경조건결정 커버리지
    * 개별 조건식이 다른 개별 조건식에 영향을 받지 않고 전체 조건식에 독립적으로 영향을 주게 하여, 조건결정 커버리지를 향상
* 다중조건 커버리지
    * 
* 기본 경로 커버리지
* 제어 흐름 테스트
* 데이터 흐름 테스트
<br>


<br>

#### 소프트웨어 테스트 케이스 작성

소프트웨어가 사용자의 요구사항을 준수하는지 확인
입력값, 실행조건, 기대결과로 구성

이따가 엄청 자세히 알아볼 것임

<br>

#### 소프트웨어 테스트 스크립트 작성

테스트 케이스의 실행 순서를 작성
= test step, test procedure

(테스트 스크립트 001) 렌터카 예약 웹 사이트 접속
도시 탭 클릭 -> 차종 선택 -> 차량 대 수 선택 -> 렌트일, 반납일, 렌트 기간 자동 계산 및 금액, 할인율 -> 예약 완료 버튼 클릭

<br>

#### 소프트웨어 테스트 시나리오 작성

사용자 관점에서 바라보았을 때, 테스트가 필요한 기능, 특징을 작성
당~연히 여러 개의 테스트 케이스를 포함

(테스트 시나리오 001) 두 도시를 방문하여 렌터카 예약
* 자동차 2대 렌트
* 2주일 이상 렌트하여 30% 할인

<br>

#### 테스트 레벨

* 함께 편성되고 관리되는 테스트 활동의 그룹 (프로젝트 책임과 연관)
* 각각의 테스트 레벨은 서로 독립적임 (책임 전과될 일은 없겠군.)

<br>

* 종류
    * 소프트웨어 개발 단계에 따라 분류한다.

    <br>

    * (단위 테스트) 사용자 요구사항에 대한 단위 모듈, 서브루틴 등을 테스트
        * 자료 구조 테스트
        * 실행 경로 테스트
        * 오류 처리 테스트
        * 인터페이스 테스트
    * (통합 테스트) 단위 테스트 통과한 모듈 사이의 인터페이스, 통합된 컴포넌트 간의 상호 작용을 검증
        * 빅뱅 테스트
        * 샌드위치 테스트
        * 상향식 테스트
        * 하향식 테스트
    * (시스템 테스트) 통합된 단위 시스템의 기능이 시스템에서 정상적으로 수행되는지 검증
        * 기능 요구사항 테스트
        * 비기능 요구사항 테스트
    * (인수 테스트) 계약상의 요구사항이 만족되었는지 확인
        * 계약 인수
        * 규정 인수
        * 사용자 인수
        * 운영상의 인수
        * 알파, 베타 테스트
